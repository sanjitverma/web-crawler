I don't see any problems at all. The classes that are truly common
(i.e. shared between your client code and server code) goes into a
shared library. This is one of the modules and let's call it "common".
Then you put the client specific code in a separate module, which has
a dependency to "common". And then you put the server specific code in
the server module, which also has a dependency to "common". Thus,
three modules which are handled by one aggregating pom. This is what
we call a multi-module build and this is standard procedure for most
dev projects.

/Anders
