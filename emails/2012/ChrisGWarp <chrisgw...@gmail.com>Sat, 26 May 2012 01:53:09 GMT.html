The simple initial answer, is that maven does not have the libraries that are
a part of the ESB or Process Server runtime library available to it.

If you view your project, in the Package Explorer view, you will see the ESB
runtime library.

All of those jars are on the compile time/runtime classpath. So this is why
your code runs and compiles in WID, but not in Maven.

This applies to any of the IBM packaged eclipses. For WAS, WESB and WPS
(Process Server and Portal Server). Every time you have a runtime library as
part of a server runtime, WID/Eclipse knows about it, but Maven does not.

So, the question becomes, how do you fix this?

My solution has always been to add a single pom file into the depdencies
that lists *all* of jars in the runtime library. Then you have to add them
all into your repo, one at a time. Painful yes, but this only needs to be
done once.

This is the approach that I've successfully used for WebSphere Application
Server, ESB, Process Server and Portal Server. I perfer to stick to the base
product version (eg 7.0.0.0) as that does not change with the various
versions of WID etc (you don't get a 7.0.0.3 runtime librarby in eclipse,
you get 7.0.0.0)

Now, having said that, this example uses 7.0.0.3 (as that is all I had been
given).

Here is a pom, that lists all of the jars that comes with the WPS 7.0.0.3
(ESB will be similar). I've named it with an artefact id of
"runtime-library", as that does not appear to clash with any real jars (so
far).

I strongly recommend that you only use 7.0.0.0 and not 7.0.0.3 (as given in
the above example). There is one execption to this, that I've noticed in
some of the Portal runtimes, new jars and functionality being introduced in
fixpacks. In those cases, you will be forced to have a 6.1.0.0 and a
6.1.0.11 runtime-library.pom file to cater for the differing dependencies.

Google the was6-maven-plugin, and you'll see that I added the servicedeploy
task to the plugin, and how I use it.

There is one final gotcha, servidedeploy is a subset of WID. There are cases
where servicedeploy can not generate the code that WID can. In this case,
you have two choices:
1. Rewrite the code, so that this does not happen.
2. Check in the generated code (bad and ugly, but it works).

I've used both approaches to deliver hundreds of Process Server modules
(ears), just like ESB reliably.
